# 误报案例弹窗统一操作设计

## 背景

当前“误报案例”区域已经有一个 `查看全部误报案例` 弹窗入口，但主页仍直接渲染：

- `待确认误报` 列表
- `已沉淀误报案例` 列表

同时，误报案例卡片上的核心操作也仍挂在这些列表里：

- `标记已确认`
- `删除无效样本`

这会带来两个问题：

1. 页面仍然偏长，和“列表都集中到弹窗里处理”的目标不一致
2. 操作入口分散在主页和弹窗两处，交互心智不统一

## 目标

把误报案例相关操作统一收进弹窗里处理，并让主页只保留摘要和入口。

成功标准：

- 主页面不再直接渲染可操作的误报案例列表
- `标记已确认` 和 `删除无效样本` 只能在误报案例弹窗里执行
- 弹窗内执行操作后原地刷新，不强制关闭
- 主页保留清晰的数量摘要和进入弹窗的入口

## 非目标

本次不做以下内容：

- 不改误报案例的数据结构或接口
- 不改误报案例卡片本身的信息排布
- 不新增第二套误报案例弹窗
- 不改“记录为误报案例”的录入弹窗
- 不调整违规反馈区或学习样本记录列表

## 方案

### 1. 主页改成摘要入口

主页“误报案例”区域不再渲染：

- `false-positive-pending-list`
- `false-positive-history-list`

改为只显示：

- 一条摘要文案
- 一个 `查看全部误报案例` 按钮

摘要文案推荐格式：

- `待确认 X 条，已沉淀 Y 条`

如果没有误报案例：

- 按钮隐藏
- 摘要显示空态，例如 `当前没有误报案例`

### 2. 所有误报操作统一进弹窗

现有 `false-positive-list` 弹窗继续作为唯一工作台。

弹窗中保留两个 section：

- `待确认误报`
- `已沉淀误报案例`

每条卡片继续复用现有 `buildFalsePositiveEntryMarkup`，并保留操作：

- 未确认项可 `标记已确认`
- 所有项可 `删除无效样本`

这样主页不再承担操作职责，弹窗成为完整的误报案例处理入口。

### 3. 弹窗内操作后原地刷新

当用户在误报案例弹窗中执行：

- `confirm-false-positive`
- `delete-false-positive`

处理完成后应：

1. 刷新 `appState.falsePositiveLog`
2. 重新渲染主页摘要
3. 如果当前弹窗仍是 `false-positive-list`，重新渲染弹窗内容

不自动关闭弹窗，方便连续处理多条。

### 4. 刷新职责收口

`renderFalsePositiveLog()` 调整为只负责三件事：

1. 更新 `appState.falsePositiveLog`
2. 更新主页摘要和入口按钮显隐
3. 如果误报案例弹窗打开中，则刷新弹窗内容

不再在主页渲染误报案例预览列表。

## 交互细节

### 主页

- 用户在主页只能看到数量摘要
- 点击 `查看全部误报案例` 进入统一弹窗

### 弹窗

- 弹窗内可以完整查看误报案例
- 弹窗内可以完成确认和删除
- 执行后列表原地刷新，继续停留在弹窗

### 空态

- 无误报案例时主页显示空态摘要
- 弹窗内各 section 仍保留各自空态文案

## 技术实现边界

改动集中在：

- `web/index.html`
- `web/app.js`
- 相关前端测试文件

实现策略：

- 复用现有 `false-positive-list` modal kind
- 复用现有 `buildFalsePositiveEntryMarkup`
- 新增主页摘要节点
- 删除主页误报案例预览列表渲染逻辑

不改后端接口。

## 风险与控制

### 风险 1：主页和弹窗状态不同步

控制：

- 所有误报列表刷新都统一走 `renderFalsePositiveLog()`
- 弹窗打开时由同一个刷新函数负责重新渲染

### 风险 2：误报操作执行后弹窗意外关闭

控制：

- 保持当前 modal kind 为 `false-positive-list`
- 操作完成后只刷新内容，不调用关闭逻辑

### 风险 3：测试仍依赖旧主页列表节点

控制：

- 更新前端测试，从“主页存在误报列表”改为“主页存在摘要入口，弹窗承担完整操作”

## 测试重点

需要验证：

1. 主页不再直接渲染待确认/历史误报列表
2. 主页显示误报数量摘要
3. 无数据时按钮隐藏、空态摘要正确
4. 点击 `查看全部误报案例` 能打开弹窗
5. 弹窗内仍能展示待确认与已沉淀两组列表
6. 弹窗内仍保留 `标记已确认` 与 `删除无效样本`
7. 在弹窗内执行操作后，弹窗保持打开并刷新内容
