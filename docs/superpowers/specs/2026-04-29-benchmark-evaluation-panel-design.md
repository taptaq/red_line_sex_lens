# 基准评测面板设计

## 背景与目标

当前系统已经有：

- 成功样本
- 误报样本
- 改写样本
- 固定评测集文件与 CLI 评测入口

但 `review-benchmark` 仍然只存在于文件和命令行层，缺少一个稳定、低门槛的人工录入入口。这样会导致两类问题：

1. 评测集扩充成本偏高，必须直接改 JSON 文件。
2. `success-samples` 和 `review-benchmark` 的职责容易混淆，用户不容易建立“学习样本库”和“考试题库”是两套东西的心智。

本设计的目标是新增一个轻量、清晰的 `基准评测` 面板，让用户可以在页面里逐条录入 benchmark 样本，并直接运行评测查看结果。

本设计不追求批量导入、趋势图表或自动从其他样本库生成 benchmark，只解决“人工逐条维护 + 直接运行校准”的第一版体验。

## 范围

本次设计覆盖：

- 在 `数据维护台` 中新增 `基准评测` tab
- 手动逐条录入基准样本
- 展示当前 benchmark 列表
- 删除单条 benchmark
- 在面板内直接运行 benchmark 评测
- 展示本次评测摘要和未匹配样本
- 把 benchmark 相关用户可见文案改成中文

本次设计不覆盖：

- 批量导入 benchmark
- 从成功样本/误报样本一键转 benchmark
- benchmark 编辑历史
- 图表化趋势统计
- 自动推荐 benchmark 标签

## 信息架构

新增入口不放在页面顶层，而是放进现有的 `数据维护台`。

原因：

- benchmark 本质是样本维护，不是新的主工作流
- 放在 `成功样本 / 误报样本 / 改写样本` 同层最符合心智
- 避免顶层 workspace 再增加一块重量级区域，导致主页面发散

具体位置：

- `数据维护台`
- 新增 tab：`基准评测`

它与现有 tab 的关系：

- `成功样本`：用于生成参考和风格画像
- `误报样本`：用于规则降权和白名单候选
- `基准评测`：用于回归验证和规则校准

## 交互设计

### 面板结构

`基准评测` tab 第一版只包含三块：

1. `录入表单`
2. `样本列表`
3. `评测结果`

整体风格应保持和当前 `成功样本`、`误报样本` 面板一致，不引入新的布局语言。

### 录入表单

每条 benchmark 只保留 4 个字段：

- 标题
- 正文
- 标签
- 预期类型

不增加备注字段，不增加来源字段，不增加复杂配置。

预期类型在界面上使用中文显示：

- `违规样本`
- `误报样本`
- `正常通过样本`

内部数据仍保持英文值，建议沿用当前 harness 约定：

- `violation`
- `false_positive`
- `success`

这样代码层更稳定，页面文案也自然。

提交成功后：

- 表单清空
- 列表立即刷新
- 结果区显示简短成功提示

### 样本列表

列表展示当前 benchmark 文件中的全部样本，至少显示：

- 标题
- 正文摘要
- 标签
- 预期类型（中文）

每条样本提供一个删除按钮。

第一版不提供行内编辑。原因是：

- 当前主要目标是先把录入链打通
- 编辑能力会带来额外状态同步和测试成本
- 删除后重录对于当前阶段足够

### 评测结果

面板内提供一个 `运行基准评测` 按钮。

点击后调用后端 benchmark harness，并在同一面板下方显示结果。

第一版结果展示只保留高价值信息：

- 总样本数
- 匹配数
- 未匹配数
- 按预期类型分布
- 按实际 verdict 分布
- 未匹配样本列表

未匹配样本列表至少展示：

- 标题或样本 ID
- 预期类型
- 实际 verdict

不在第一版里展示过深的调试信息，避免面板变成日志阅读器。

## 数据设计

benchmark 样本文件继续使用现有路径：

- `data/evals/review-benchmark.json`

每条样本结构建议保持简单：

```json
{
  "id": "benchmark-1714380000000",
  "expectedType": "false_positive",
  "input": {
    "title": "关系沟通提醒",
    "body": "这篇内容只是分享边界感和沟通方式。",
    "tags": ["关系", "沟通"]
  }
}
```

说明：

- `id` 自动生成
- `expectedType` 保持英文代码值
- `input` 结构直接复用当前分析链可接受的输入格式

这样能够减少转换逻辑，也更方便直接喂给现有 harness。

## 后端设计

建议新增一组轻量 API：

- `GET /api/review-benchmark`
  返回当前 benchmark 列表
- `POST /api/review-benchmark`
  新增一条 benchmark 样本
- `DELETE /api/review-benchmark`
  按 `id` 删除一条样本
- `POST /api/review-benchmark/run`
  运行当前 benchmark harness 并返回摘要

这些 API 只围绕现有 `review-benchmark.json` 文件做读写，不引入数据库，不引入复杂状态。

后端职责边界：

- `data-store` 负责 benchmark 文件读写和基础归一化
- `server.js` 负责 API 路由
- 现有 `src/evals/review-benchmark-harness.js` 继续负责评测执行

## 前端设计

前端修改集中在：

- `web/index.html`
- `web/app.js`
- `web/styles.css`

建议实现方式：

- 在 `数据维护台` tab strip 中新增 `基准评测`
- 新建对应 panel
- 表单提交走 JSON API
- 列表和结果通过现有 `refreshAll()` 或新增局部刷新函数更新

界面文案上，用户可见内容尽量全部中文：

- `违规样本`
- `误报样本`
- `正常通过样本`
- `运行基准评测`
- `当前没有基准样本`
- `基准样本已保存`

代码内部可继续保留英文枚举值，避免大范围联动改名。

## 错误处理

第一版需要处理的错误场景：

1. 提交空正文或空标题
   返回明确提示，不写入文件。
2. 预期类型非法
   后端归一化到默认值，或直接拒绝并提示。
3. benchmark 文件为空
   运行评测时提示“当前没有基准样本”。
4. 删除不存在的 ID
   返回 404 或无副作用成功响应，但前端应提示该样本不存在。
5. 文件读写失败
   页面展示简短错误信息，不吞掉失败。

## 测试策略

第一版至少补这几类测试：

- 数据层：
  - benchmark 样本读写、归一化、删除
- API：
  - 创建、列表、删除、运行评测
- 前端：
  - 新 tab 是否存在
  - 表单是否提交成功
  - 结果区是否渲染摘要
- 回归：
  - 现有 `eval-review-benchmark` harness 仍可从 CLI 运行

## 验收标准

- 用户可以在页面中手动录入一条 benchmark 样本
- 用户可以在页面中看到 benchmark 样本列表
- 用户可以删除单条 benchmark 样本
- 用户可以在页面中点击按钮运行 benchmark 评测
- 页面会显示样本总数、匹配/未匹配数和未匹配样本列表
- benchmark 相关用户可见文案使用中文
- 不影响现有成功样本、误报样本和生成工作台功能
