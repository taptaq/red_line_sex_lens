# 误报回流链路设计

## 目标

给当前检测系统补上一条与“违规回流”对称的“误报回流”链路，用来记录这类样本：

- 系统检测结果为 `manual_review` / `hard_block`
- 但内容实际发到小红书后未被处罚

这条链路的目标不是自动放松规则，而是沉淀“规则可能偏严”的证据，帮助后续人工复核并迭代规则。

## 证据分级

误报样本分成两类，并赋予不同可信度：

- `platform_passed_pending`
  - 用户手动记录“已发出，目前正常”
  - 这是弱证据，只用于提示，不直接用于规则松绑
- `platform_passed_confirmed`
  - 用户补充或更新为“经过观察期仍正常”
  - 这是强证据，可进入规则偏严复盘统计

当前产品语义上，两类都要保存，但所有“规则调整建议”默认只优先参考 `confirmed` 样本。

## 数据模型

新增文件：

- `data/false-positive-log.json`

单条记录建议字段：

- `id`
- `createdAt`
- `updatedAt`
- `status`
  - `platform_passed_pending`
  - `platform_passed_confirmed`
- `observedAt`
  - 用户确认“观察期后仍正常”的时间
- `observationWindowHours`
  - 例如 `24` / `48`
- `title`
- `body`
- `coverText`
- `tags`
- `analysisSnapshot`
  - 记录当时系统 verdict / score / topHits / categories / suggestions
- `crossReviewSnapshot`
  - 可选，保存当时交叉复判摘要
- `userNotes`
  - 用户补充说明
- `publishMeta`
  - 可选，如发布时间、渠道备注
- `falsePositiveAudit`
  - 对当前规则“可能偏严”的结构化结论

## 审核信号

新增 `falsePositiveAudit`，核心信号定义如下：

- `strict_pending`
  - 系统结论偏严，但样本仍在观察期
- `strict_confirmed`
  - 系统结论偏严，且已有观察期确认，是强证据
- `not_enough_evidence`
  - 文本或状态不足以支持判断

这层信号只负责“样本解释”，不直接改词库。

## 前端入口

在“规则检测”或“合规改写”结果区域新增一个显式入口：

- `记录为误报样本`

点击后自动带入当前：

- 标题
- 正文
- 封面文案
- 标签
- 当前规则结论
- 命中项
- 可选的交叉复判摘要

用户只需要补：

- 当前状态：`已发出，目前正常` / `观察期后仍正常`
- 观察时长
- 备注

同时，在数据维护区新增一个“误报样本”分区，用于：

- 查看误报日志
- 将 `pending` 更新为 `confirmed`
- 删除明显无效样本

## 布局与展示要求

这次功能实现必须把布局稳定性当成硬要求，而不是附带优化。

前端界面需要满足：

- 新增入口和误报样本分区不能造成现有面板错位、卡片拉伸异常或模块高度失衡
- 长文本字段（正文、命中原因、备注、复盘说明）必须支持完整查看，不能只显示一部分就结束
- 不允许依赖“固定高度 + 隐藏溢出”掩盖问题；如需控制视觉长度，应提供显式展开全文的交互
- 桌面端和移动端都要可读，不能出现列宽过窄导致文本被异常挤压
- 新功能的卡片样式、间距、圆角、信息密度需要与现有页面视觉语言保持一致

推荐实现方式：

- 误报样本详情中的长文本统一采用“摘要 + 展开全文”或“默认展开全文阅读块”
- 新增面板优先使用按内容高度自然排布的网格，而不是强制等高卡片
- 如果新增字段较多，优先拆成主信息区和附加信息区，避免把所有信息塞进同一列

## 服务端接口

建议新增：

- `GET /api/false-positive-log`
  - 返回误报日志
- `POST /api/false-positive-log`
  - 新增误报样本
- `PATCH /api/false-positive-log`
  - 更新样本状态，例如从 `pending` 改为 `confirmed`
- `DELETE /api/admin/false-positive-log`
  - 删除误报样本

服务端处理时要统一生成：

- `analysisSnapshot`
- `falsePositiveAudit`

## 规则进化策略

误报样本不会自动删规则或自动降级。系统只做“候选提示”，真正调整仍由人工决定。

建议优先支持这三类动作：

- 将某条规则风险等级从 `manual_review` 调低到 `observe`
- 为某条规则补充白名单语境
- 缩窄某条 regex 规则命中范围

后续可以基于误报样本增加“规则偏严候选榜”，但第一版不必自动生成新的词库草稿，先把样本采集和状态确认做扎实。

## 与现有链路关系

当前系统已有：

- `违规回流`：解决漏判
- `复核队列`：解决是否入库

本方案新增：

- `误报回流`：解决偏严和误杀

三者职责保持分离，避免把“平台处罚样本”和“平台放行样本”混进同一日志，导致后续复盘语义混乱。

## 测试范围

第一阶段至少覆盖：

- 误报样本读写到 `false-positive-log.json`
- `pending` / `confirmed` 状态更新
- `falsePositiveAudit` 生成逻辑
- 前端预填当前检测结果后能成功提交
- 误报日志页面能正确展示状态与分析快照
- 误报相关新增界面在桌面端和移动端无明显错位、截断或布局溢出

## 范围边界

本次不做：

- 自动根据误报样本修改词库
- 自动生成误报版 review queue
- 平台真实发布状态的自动抓取
- 复杂统计面板

先把“记录、确认、复盘”这条闭环打通，再考虑第二阶段做规则调整建议和聚合统计。
