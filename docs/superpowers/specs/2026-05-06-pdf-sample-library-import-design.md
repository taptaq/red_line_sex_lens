# PDF 批量导入学习样本设计

## 背景

当前系统已经支持在 `学习样本` 区手动新增单条样本，并逐步补全：

- 基础内容
- 参考属性
- 生命周期属性

用户现在手头已有一批历史小红书笔记，存储为多个 PDF 文件。每个 PDF 对应一篇笔记，结构较统一：

- 第一行是标题
- 后续内容是正文

这些历史笔记希望批量导入系统，先自动回填标题和正文，再由用户手动补：

- `collectionType`
- `tags`
- `likes`
- `favorites`
- `comments`

用户明确要求：

- 输入来源改为 `PDF`，不做 Word
- 支持同时上传多个文档
- `collectionType` 不自动推断，导入后逐篇手动选择

## 目标

新增一个 `PDF 批量导入学习样本` 的最小可用能力，让用户可以：

1. 一次选择多个 PDF
2. 系统自动提取每个 PDF 的文本
3. 用 `首行标题 + 其余正文` 生成导入草稿
4. 在导入结果列表中逐条补全和修正信息
5. 仅在用户确认后，才把记录正式写入样本库

## 非目标

本次不做以下能力：

- Word / docx 导入
- OCR 图片识别型 PDF
- 从文件名或目录自动推断 `collectionType`
- 自动生成标签
- 自动根据互动数据决定参考样本等级
- 导入后自动启用 `reference.enabled`
- 批量编辑整表式高级导入工作台

第一版只处理“可直接抽取文本的 PDF”。

## 用户流程

### 1. 入口

在 `学习样本` 区现有的创建入口附近，增加一个 `批量导入 PDF` 按钮。

点击后打开文件选择器，支持一次选择多个 `.pdf` 文件。

### 2. 上传与解析

前端把所选 PDF 发送到新的导入接口。

后端逐个解析 PDF 文本：

- 去掉明显空白段
- 将第一条有效文本行作为标题
- 将剩余文本拼成正文

如果某个 PDF 无法提取出有效文本，或只提取到极短内容，则该条标记为解析失败，不直接入库。

### 3. 导入草稿列表

解析完成后，前端显示一个“导入结果列表”，每个 PDF 对应一条待确认草稿，包含：

- 文件名
- 解析出的标题
- 解析出的正文摘要
- 可编辑标题
- 可编辑正文
- 手动选择 `collectionType`
- 手动填写 `tags`
- 手动填写 `likes / favorites / comments`
- 勾选“确认导入”

如果标题或正文解析不准，用户可在这里直接修改。

### 4. 正式导入

只有被用户勾选并提交的草稿才会真正写入 `note-records.json`。

导入后的默认记录策略：

- `source = "manual"`
- `stage = "draft"`
- `sampleType = ""`
- `reference.enabled = false`
- `publish.status = "not_published"`
- `publish.metrics` 使用用户手填值，未填时为 `0`

## 数据映射

每条导入草稿最终映射到样本库记录结构：

### 基础内容

- `note.title`：PDF 首行，允许用户编辑
- `note.body`：PDF 剩余正文，允许用户编辑
- `note.coverText`：默认空
- `note.collectionType`：用户手动选择，必填
- `note.tags`：用户手动输入并拆分

### 生命周期

- `publish.status`：默认 `not_published`
- `publish.metrics.likes`：用户手填，默认 `0`
- `publish.metrics.favorites`：用户手填，默认 `0`
- `publish.metrics.comments`：用户手填，默认 `0`

### 参考属性

- `reference.enabled = false`
- `reference.tier = ""`
- `reference.selectedBy = ""`
- `reference.notes = ""`

## 解析规则

为保持第一版稳定，PDF 解析规则尽量简单明确：

1. 将 PDF 提取出的文本按行拆分
2. 去除首尾空格后，过滤空行
3. 第一条有效行作为标题
4. 剩余行使用换行拼接成正文
5. 如果没有正文，允许保留为空，但该条在前端需提示用户补正文后再确认导入

### 失败判定

以下情况视为解析失败或需人工修正：

- 完全提取不到文本
- 只有标题，没有正文
- 文本明显过短，无法构成有效笔记内容

前端应明确显示失败原因，用户可跳过这些条目。

## 前端设计

### 样本库区新增内容

在现有 `新增学习样本` 按钮附近增加：

- `批量导入 PDF` 按钮

并新增一个导入结果面板，用于展示本次解析出的待确认草稿列表。

### 每条草稿的交互项

每条草稿至少包含：

- 文件名
- 标题输入框
- 正文输入框
- `collectionType` 下拉
- `tags` 输入框
- `likes / favorites / comments` 数字输入框
- “确认导入”复选框
- 解析状态或错误提示

### 提交策略

用户点击“导入已确认条目”后：

- 只提交勾选的有效条目
- 未勾选条目不写入样本库
- 校验不通过条目阻止提交并显示原因

## 后端设计

### 新增接口

新增两段能力：

1. `PDF 解析接口`
   接收多个 PDF 文件，返回导入草稿列表
2. `导入确认接口`
   接收用户修正后的草稿列表，逐条写入样本库

这样可以把“解析”和“正式入库”拆开，避免一上传就污染正式数据。

### 去重策略

正式写入时继续复用现有 `note-records` 指纹逻辑：

- 标题
- 正文
- 封面文案
- 标签

如果同一内容重复导入，应走已有的合并 / 去重策略，而不是无脑重复新增。

## 校验规则

导入确认前，每条记录至少满足：

- `title` 非空
- `body` 非空
- `collectionType` 非空

互动数据允许为空，提交时按 `0` 处理。

`tags` 允许为空，但建议填写。

## 错误处理

### 文件级错误

单个 PDF 解析失败时：

- 不影响同批次其他文件
- 在该条草稿上展示失败原因

### 提交级错误

如果正式导入时某些条目失败：

- 返回逐条结果
- 成功和失败分开提示
- 不因为一条失败导致整批全部回滚

## 验证与测试

需要覆盖以下测试面：

1. 后端 PDF 解析逻辑
   - 多文件输入
   - 首行标题识别
   - 正文拼接
   - 空文件 / 短文本失败

2. 前端导入草稿交互
   - 渲染导入结果列表
   - 可编辑标题 / 正文
   - `collectionType` 必填校验
   - 只提交勾选条目

3. 样本库写入
   - 正确映射到 `note-records`
   - 默认 `reference.enabled = false`
   - 默认 `publish.status = not_published`
   - 指标数据正确写入

## 预期结果

实现完成后，用户可以把历史 PDF 笔记快速转成学习样本，并在较低人工成本下开始沉淀：

- 真实旧内容
- 真实集合类型
- 手动补齐的标签
- 手动补齐的互动数据

为后续：

- 风格画像
- 生成参考样本
- 生命周期复盘

建立更真实的数据基础。
