# AI 记忆共享层设计

## 背景

当前系统已经具备较完整的样本与反馈沉淀基础：

- `note-records` 作为统一样本主存储
- `feedback.log` 记录真实违规反馈
- `false-positive-log` 记录误报与过严案例
- `reference`、`sampleWeight`、`calibration` 已经承担部分经验筛选与运行时参考职责
- 生成链路与校验链路都已经开始消费“参考样本”这一类经验资产

但系统目前仍有两个明显缺口：

1. `召回能力不够智能`
   当前更偏结构化筛选和轻量相似支持，难以在样本量继续增长后稳定召回“真正相关”的历史经验。
2. `经验沉淀仍偏事实存储`
   反馈、误报、成功样本虽然被记录下来，但还没有形成一层可复用、可治理、可解释的“经验记忆”。

用户希望第一版 AI 记忆同时服务两条在线链路：

- 内容检测
- 改写 / 生成

同时明确选择以下边界：

- 采用 `共享记忆层`
- 写入采用 `半自动`
- 底座采用 `本地向量库`
- embedding 采用 `本地 / open-source` 路线

## 目标

建立一套项目内可控的 AI 记忆共享层，同时提升：

1. `召回更准`
   在检测、改写、生成时更准确地找回相似违规样本、误报反例、成功样本和风格经验。
2. `经验能沉淀`
   把高置信度反馈与样本提炼成可复用的结构化记忆卡片。
3. `运行时可解释`
   记忆层参与检测、改写、生成时，必须能说明来源与作用，不变成黑箱。
4. `治理可控`
   错误记忆、过时记忆、低置信度结论不能轻易污染在线召回。

## 非目标

第一版不做以下能力：

- 用记忆层替代当前规则引擎
- 让记忆层直接决定 `hard_block` 放行
- 自动修改正式规则词库或白名单
- 引入跨账号共享且未隔离的长期记忆
- 一开始就做知识图谱主导的多跳推理
- 引入外部通用 AI memory 平台作为主数据层

## 总体方案

第一版采用 `事实层 + 向量检索层 + 记忆卡片层 + 运行时编排层 + 人工治理层` 的五层结构。

### 1. 事实层

继续保留现有 JSON 数据结构作为唯一事实源，包括但不限于：

- `note-records`
- `feedback.log`
- `false-positive-log`
- `style-profile`
- 现有 calibration / lifecycle / reference 相关字段

事实层不被向量库替代。所有检索文档与记忆卡片都必须能追溯回事实层。

### 2. 向量检索层

新增项目内本地向量索引，用于保存可检索文档的 embedding 与检索元数据。其职责是：

- 为检测、改写、生成提供语义召回能力
- 为后续记忆卡片提炼提供候选聚类与证据支持
- 与现有 `sampleWeight`、`reference`、`publish status` 一起参与重排

### 3. 记忆卡片层

在原始事实之上提炼结构化经验，沉淀为可治理的“记忆卡片”，用于：

- 风险解释
- 误报边界说明
- 改写策略提示
- 风格与结构经验提示

记忆卡片不是原始事实，必须单独存储、单独治理、单独召回。

### 4. 运行时编排层

检测、改写、生成三条链路不直接裸查向量库，而是统一通过 `memory retrieval service`：

1. 元数据过滤
2. 向量召回
3. 业务重排
4. 上下文打包

### 5. 人工治理层

对自动写入、激活、降权、失效做治理，避免脏记忆污染系统。

## 数据分层

### 一、检索文档

检索文档是对原始事实的忠实映射，主要为向量召回服务。建议纳入以下几类：

1. `参考样本文档`
   来源于 `reference.enabled === true` 且满足当前达标口径的样本。
2. `普通样本文档`
   来源于已入样本库但未进入参考池的样本，默认权重较低。
3. `违规反馈文档`
   来源于真实处罚反馈，包含平台原因、笔记正文、时间与标签等信息。
4. `误报文档`
   来源于已确认或高置信误报记录。
5. `改写成功文档`
   来源于最终通过或表现较好的改写结果，可作为“改写经验源”。

检索文档保留更多事实细节，但不直接等于给模型的最终上下文。

### 二、记忆卡片

记忆卡片是从多个事实中提炼出的经验结论。设计目标上，第一版记忆卡片预留以下四类：

1. `风险模式卡`
   描述某类平台原因、命中组合、语境模式之间的稳定关联。
2. `误报反例卡`
   描述哪些表达单独出现通常安全，哪些组合场景下才危险。
3. `改写策略卡`
   描述某类风险通常用什么改法更稳、更自然。
4. `风格经验卡`
   描述某合集类型、账号风格、结构习惯与高表现表达方式。

当前实现状态：

- 已实现候选与激活规则的卡片生成逻辑，首版实际落地的是 `风险模式卡`。
- `改写策略卡`、`误报反例卡`、`风格经验卡`、`风险边界卡` 仍属于运行时可消费的卡片种类约定，是否有数据取决于后续人工治理或外部写入流程。

记忆卡片必须显式区分于原始检索文档，不能混存混排。

## 写入与半自动入库规则

### 一、正式检索层的自动入库对象

以下内容可半自动进入正式检索层：

- 已启用且达标的参考样本
- 已确认误报
- 有明确平台处罚结果的违规反馈
- 生命周期与发布状态清晰的样本记录
- 已确认的改写成功记录

### 二、候选层对象

以下内容先进入候选层，待人工确认后再进入正式层：

- 普通样本
- 模型提炼出的风险模式卡
- 模型提炼出的改写策略卡
- 仅由单一弱证据支撑的记忆卡片

### 三、禁止自动进入

以下内容不得自动进入正式层：

- 只有模型猜测、没有原始证据的总结
- 无法回溯 `sourceIds` 的记忆
- 多个来源互相冲突的经验结论
- 任何试图放宽 `hard_block` 的记忆结论

### 四、治理字段

每条检索文档与记忆卡片都必须至少携带以下字段：

- `id`
- `kind`
- `status`
- `confidence`
- `sourceQuality`
- `sourceIds`
- `createdAt`
- `updatedAt`
- `accountScope`
- `collectionType`
- `riskCategories`
- `retrievalWeight`
- `embeddingVersion`

其中第一版建议的 `status` 至少包含：

- `candidate`
- `active`
- `suppressed`
- `archived`

当前实现状态：

- 首版运行时与 CLI 已实际使用 `candidate`、`active`。
- `suppressed`、`archived` 作为治理预留状态保留在设计模型中，但尚未形成完整的降权、归档与运营闭环。

## 运行时检索设计

三条在线链路统一走共享记忆层，但采用不同的召回配方。

### 一、检测链路

检测链路中，记忆层的职责是：

1. 增强风险解释
2. 辅助识别误报或过严判断
3. 触发待审核规则候选

检测时优先召回以下桶：

- `风险反馈桶`
- `误报反例桶`
- `参考样本桶`
- `风险模式卡桶`

检测链路允许：

- 增强解释
- 辅助 `manual_review -> observe`
- 生成待审核规则候选

检测链路明确禁止：

- 通过记忆层让 `hard_block -> pass`
- 让普通样本或弱证据卡片直接放宽高风险结果

### 二、改写链路

改写链路的目标是按历史经验做“定向修”，而不是只依赖当前单条稿子的即时推理。

改写时优先召回：

- 相似违规样本
- 改写成功样本
- 改写策略卡
- 误报反例

给模型的上下文应整理为：

1. `相似失败原因`
2. `可借鉴的成功改法`
3. `不要过度收紧的边界提醒`

当前实现状态：

- 改写链路已经接入共享 `memory retrieval service`。
- 首版会把 `riskFeedback`、`falsePositiveHints`、`referenceSamples` 和 `memoryCards` 整理成结构化提示块后喂给改写 / patch prompt，而不是直接拼接大量原始样本文本。
- 首版 `memoryCards` 检索桶按 `rewrite_strategy_card` 过滤；是否召回到结果取决于当前卡片数据是否存在。

第一版不建议简单拼接大量原文样本给模型，优先输出整理后的上下文块。

### 三、生成链路

生成链路的重点是：

- 风格贴合
- 安全边界
- 成功表达复用

生成时优先召回：

- 成功参考样本
- 风格经验卡
- 同合集类型样本
- 风险边界卡

当前实现状态：

- 生成链路已经接入共享 `memory retrieval service`。
- 首版运行时实际消费 `reference_sample`，以及 `style_experience_card` / `risk_boundary_card` 两类记忆卡。
- 首版不会把原始违规反馈原文直接喂给生成模型。

第一版生成链路中，不应直接向模型喂入原始违规反馈原文。违规反馈应先被提炼为风险边界提示后再参与生成。

### 四、统一检索流程

三条链路统一采用以下四步：

1. `metadata filter`
   按 `kind / status / accountScope / collectionType / riskCategories` 过滤候选。
2. `vector recall`
   在过滤结果中进行向量召回。
3. `business rerank`
   结合 `sampleWeight`、`reference.enabled`、`publish status`、`confidence` 等业务信号重排。
4. `context packing`
   将召回结果整理成链路可消费的结构化上下文，而不是直接暴露裸文档列表。

第一版建议的统一返回格式如下：

```json
{
  "riskFeedback": [],
  "falsePositiveHints": [],
  "referenceSamples": [],
  "memoryCards": [],
  "retrievalMeta": {
    "queryKind": "analysis",
    "embeddingVersion": "local-embedding-v1",
    "candidateCount": 0
  }
}
```

## 本地向量库与本地 embedding 设计

### 一、主存储与索引存储分离

主存储继续使用现有 JSON 文件。向量索引建议新增独立目录，例如：

```text
data/memory/
  documents.jsonl
  cards.jsonl
  embeddings.jsonl
  index-meta.json
```

向量索引不是事实源，只是运行时加速层与检索层。

### 二、模块拆分

建议新增以下模块：

- `src/memory/memory-document.js`
- `src/memory/memory-card.js`
- `src/memory/embedding-provider.js`
- `src/memory/vector-store.js`
- `src/memory/retrieval-service.js`

职责分别对应：

- 事实转检索文档
- 经验卡片生成与校验
- 本地 embedding 提供者封装
- 向量写入、更新、检索
- 面向业务链路的统一检索服务

### 三、索引更新策略

第一版采用 `写时增量标脏 + 异步小批重建 + 定时全量重建` 的策略。

具体建议：

1. 数据更新时，标记相关文档为 `dirty`
2. 后台异步重建该批文档 embedding
3. 提供全量重建命令，例如 `npm run memory:rebuild`
4. 提供当前索引 / 卡片摘要命令，例如 `npm run memory:inspect`
5. embedding 模型切换时，通过 `embeddingVersion` 触发全量重建

第一版优先保证：

- 可恢复
- 可重建
- 可验证

不追求重型实时索引系统。

### 四、记忆卡片激活规则

记忆卡片采用两段式流转：

1. `候选生成`
   新反馈、新误报、新成功样本进入后，先生成候选卡片。
2. `晋升激活`
   满足条件时才进入 `active` 状态。

第一版建议的激活条件：

- 至少有 2 条来源支撑，或
- 有 1 条高置信人工确认来源
- 且不存在明显冲突证据

## 召回污染防护

第一版必须实现以下三道护栏：

### 1. 事实与推论分仓

- 原始样本和反馈属于事实
- 风险模式、改写策略、风格经验属于推论

两者分仓存储、分仓检索、分仓重排。

### 2. 召回分桶

不同链路消费不同的记忆桶：

- 检测优先风险反馈、误报反例、风险模式卡
- 改写优先违规样本、改写成功样本、改写策略卡
- 生成优先成功样本、风格经验卡、风险边界卡

不允许把所有记忆对象直接混成一个上下文。

### 3. 降权与失效

当某条记忆长期被新反馈打脸、人工明确否定、或已明显过时，应支持：

- 降权
- `suppressed`
- `archived`

被抑制或归档的对象不应继续参与在线召回。

当前实现状态：

- `suppressed` / `archived` 仍属于治理目标，首版尚未补齐完整的在线降权、失效流转与治理工具。

## 验证与评估

第一版至少验证以下四类问题：

### 1. 召回质量

验证是否能比当前机制更准确召回：

- 相似违规反馈
- 相似误报反例
- 同风格成功样本
- 有效改写经验

### 2. 链路收益

验证共享记忆层是否带来实际收益：

- 检测解释是否更清楚
- 误报识别是否更稳
- 改写是否更少过度收紧
- 生成是否更贴风格且更稳

### 3. 污染控制

验证错误记忆、弱证据记忆、过时记忆不会轻易进入正式层并污染在线召回。

### 4. 性能

验证在当前项目样本量下：

- 本地 embedding 可接受
- 本地召回可接受
- 索引重建可接受

### 五、离线评估建议

建议新增一组 memory benchmark：

- 检测侧 benchmark：给定内容后，检查是否召回应命中的违规/误报证据
- 生成侧 benchmark：给定 brief 后，检查是否召回同类型高质量参考样本
- 改写侧 benchmark：给定风险内容后，检查是否召回合适的改写经验

建议记录的指标包括：

- top-k 命中率
- 人工满意度
- 改写后通过率变化
- 检测解释可用性反馈

## 分阶段落地建议

### Phase 1：共享检索底座

- 建立检索文档格式
- 建立本地 embedding provider 抽象
- 建立本地向量索引
- 打通检测 / 改写 / 生成统一 retrieval service

### Phase 2：候选记忆卡片

- 从反馈、误报、成功样本中生成候选卡片
- 引入 `candidate / active / suppressed / archived` 流转
- 先只在检测解释、改写提示和生成提示中使用 active 卡片

当前实现状态：

- 已完成 `candidate -> active` 的首版激活逻辑。
- 首版候选卡片生成逻辑聚焦 `风险模式卡`。
- `suppressed / archived` 仍保留为后续治理阶段能力。

### Phase 3：评估与治理

- 增加 benchmark 与回归验证
- 增加记忆降权与失效能力
- 增加 memory rebuild / inspect / audit 工具

当前实现状态：

- 已补齐 `memory:rebuild`、`memory:inspect` 命令。
- `audit`、降权与失效治理仍未落地。

## 总结

第一版 AI 记忆不应被设计成通用聊天记忆，也不应被外部 memory 产品替代。更适合当前项目的方案是：

在现有 JSON 事实层之上，新增一套 `本地 embedding + 本地向量索引 + 记忆卡片层 + 统一检索服务` 的共享记忆层，采用半自动写入，自动激活高置信事实，审核后激活推论型记忆，同时服务检测、改写与生成三条在线链路。首版已完成共享检索底座、检测 / 改写 / 生成接入、`memory:rebuild` / `memory:inspect` 命令以及 `风险模式卡` 的候选与激活逻辑；更完整的卡片类型治理、`suppressed / archived` 流转和 audit 能力仍在后续阶段。
