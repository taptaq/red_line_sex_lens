# 样本库长列表弹窗化设计

## 背景

当前样本库页同时在主页面直接渲染：

- 误报案例列表
- 记录列表

随着误报样本和学习样本记录增长，这两块会显著拉长页面滚动距离，影响主工作流的可达性，也让右侧详情区更难聚焦。

项目里已经有可复用的统一弹窗体系：

- `sample-library-modal`

因此这次改动目标不是重做信息架构，而是在不破坏现有操作路径的前提下，把长列表收起来。

## 目标

把以下两块改成“短预览 + 查看全部弹窗”模式：

1. `误报案例`
2. `记录列表`

成功标准：

- 主页面默认滚动明显缩短
- 用户仍能快速看到最近几条记录
- 点开弹窗后仍可完成原有主要操作
- 不重做样本详情编辑流，不引入新的视觉体系

## 非目标

本次不做以下内容：

- 不改样本详情区的整体结构
- 不把详情编辑步骤也搬进弹窗
- 不新增新的弹窗组件体系
- 不重做误报案例卡片内容结构
- 不调整后端接口或数据格式

## 方案

### 1. 误报案例区

主页面中：

- `待确认误报`
- `已沉淀误报案例`

这两组都只保留最近 `3` 条预览。

在预览区底部增加统一入口：

- `查看全部误报案例`

点击后打开弹窗，弹窗中展示完整误报案例列表，并保留已有操作能力，例如：

- 确认误报
- 删除误报

主页面仍然保留空态提示；如果没有数据，不展示“查看全部”按钮。

### 2. 记录列表区

样本库左侧的长记录列表改成：

- 默认只显示最近 `3` 条预览
- 保留当前列表计数文案
- 增加 `查看全部记录列表` 按钮

点击后打开弹窗，弹窗中展示当前筛选条件下的完整记录列表。

弹窗内的记录项仍然支持：

- 打开某条记录
- 高亮当前选中记录

选择记录后：

- 关闭弹窗
- 回到主页面右侧详情区
- 保持现有详情编辑工作流不变

### 3. 弹窗复用策略

直接复用现有 `sample-library-modal`：

- 新增两个 modal kind
  - `false-positive-list`
  - `sample-library-record-list`

不新增新的 overlay、dialog、footer 体系，只补充少量内容区样式。

### 4. 预览规则

预览统一采用最近 `3` 条：

- 按当前页面已有排序逻辑截断
- 不改变完整列表的排序逻辑

这样可以保证：

- 主页面短而稳定
- 最新信息仍然可见
- 弹窗和主页面的认知保持一致

## 交互细节

### 误报案例弹窗

弹窗内分两个 section：

- 待确认误报
- 已沉淀误报案例

每个 section 使用现有误报案例卡片渲染函数，避免主页面预览和弹窗全量视图出现两套结构。

### 记录列表弹窗

弹窗内展示：

- 当前筛选结果条数
- 完整记录列表

点击记录卡片后，直接调用现有“打开记录”逻辑，避免分叉出新状态机。

### 页面滚动

主页面只保留预览内容，因此正常情况下不再因为长列表导致过深滚动。

弹窗自身内容区负责滚动：

- 主页面不滚
- 长列表在弹窗内容区滚动

## 技术实现边界

前端改动集中在：

- `web/app.js`
- `web/index.html`
- `web/styles.css`

实现策略：

- 复用现有 modal state
- 新增预览截断渲染
- 新增两个“打开全量列表弹窗”的 action
- 新增两个 modal markup builder

不改后端接口。

## 风险与控制

### 风险 1：主页面和弹窗出现两套不一致列表

控制：

- 误报案例继续复用现有 `buildFalsePositiveEntryMarkup`
- 记录列表继续复用现有记录卡片渲染逻辑

### 风险 2：弹窗打开记录后状态不同步

控制：

- 继续走现有 `open-sample-library-record` 逻辑
- 弹窗只负责展示和触发，不自己维护详情状态

### 风险 3：交互入口过多导致样本库更复杂

控制：

- 主页面只增加两个“查看全部”入口
- 不额外增加新 tab 或新二级导航

## 测试重点

需要验证：

1. 主页面默认只显示 `3` 条预览
2. 无数据时空态文案仍正确
3. 点击“查看全部误报案例”能打开弹窗
4. 点击“查看全部记录列表”能打开弹窗
5. 弹窗内能展示完整列表
6. 从记录列表弹窗点击记录后，详情区正确切换
7. 弹窗关闭后页面状态不乱
