# 学习样本收口与浏览数接入设计

## 背景

当前前端的信息结构已经逐步收敛到 `学习样本` 作为内容沉淀主场，但仍然存在两类问题：

1. `回流中心` 与 `学习样本` 在职责上出现了明显重叠
2. 学习样本的数据指标仍只覆盖 `点赞 / 收藏 / 评论`，缺少 `浏览数`

第一类问题会带来产品认知负担。当前用户需要在两个区域之间切换：

- `回流中心`
- `学习样本`

而两边都在处理以下相邻事务：

- 真实违规反馈
- 误报样本
- 反馈候选规则
- 学习样本沉淀
- 平台结果回填

第二类问题则导致样本质量判断不够完整。实际运营中，`浏览数` 也是非常常见的参考指标，如果不纳入，样本录入和参考样本池判断会缺少一块关键信号。

## 目标

本次设计同时解决两件事：

1. 把 `回流中心` 尽量并入 `学习样本`，让用户以后只围绕一个主场处理样本、回流和发布结果
2. 把 `浏览数` 作为第四个互动指标，统一加入学习样本相关入口与展示链路，并接入参考样本池判定逻辑

## 非目标

本次不做以下内容：

- 重做整体导航体系
- 把词库维护放进样本池弹窗
- 调整现有参考样本池的核心正向合规边界
- 重新设计生成与校验的全链路后端结构
- 让 `浏览数` 单独直接决定参考样本入池

## 设计一：回流中心并入学习样本

### 总体方向

采用 `强收口` 方案：

- 保留 `学习样本`
- 取消 `回流中心` 独立页签
- 把其内容并入 `学习样本`

目标是让用户以后默认只记住一个主处理入口：

- `学习样本`

### 新的职责分层

#### 1. 学习样本 · 日常处理

这是高频主场，负责：

- 学习样本记录列表
- 单条样本编辑
- 样本池工作台
- 待处理回流队列
- 平台结果回填
- 误报样本与反例处理

#### 2. 学习样本 · 扩展维护

这是低频维护区，负责：

- 规则候选队列
- 自定义词库
- 种子词库
- 白名单 / 误报相关维护
- 历史回放 / 系统校准

#### 3. 样本池弹窗工作台

样本池弹窗继续只负责：

- 参考样本池
- 普通样本池
- 反例样本池

不承载词库维护，不承载完整回流表单。

### 页面结构调整

#### 需要保留的能力

- `学习样本` 主页面
- `样本池工作台`
- `规则维护`
- `系统校准`
- 当前的快速违规反馈录入能力

#### 需要并入学习样本的内容

原 `回流中心` 中以下内容并入 `学习样本`：

- 待处理反馈列表
- 待确认误报
- 已处理误报历史
- “需要补规则”的候选入口

并入后的定位是：

- 学习样本中的 `回流待处理区`

#### 需要删除或降级的内容

- `回流中心` 独立 tab
- 与学习样本重复的误报处理入口
- 与学习样本重复的“去哪里处理”说明文案
- 那些本来就可以在学习样本完成的重复跳转按钮

### 用户路径

收口后的推荐路径：

1. 在内容工作台得到检测结果、改写结果或生成结果
2. 真实平台结果、违规反馈或误报案例进入 `学习样本`
3. 在 `学习样本` 中判断其角色：
   - 参考样本
   - 普通样本
   - 反例样本
   - 待补规则线索
4. 如果问题上升到规则层，再从同页进入 `扩展维护 > 规则维护`

### 词库位置

词库不放入样本池弹窗，也不保留在独立回流中心下。

词库统一放在：

- `学习样本 > 扩展维护 > 规则维护`

这样形成清晰分工：

- `学习样本 / 回流` 负责输入与判断素材
- `规则维护` 负责把确认过的规律沉淀成正式规则

## 设计二：浏览数全链路接入

### 指标角色

`浏览数` 作为第四个互动指标，与以下字段并列：

- 点赞
- 收藏
- 评论
- 浏览数

### 接入范围

采用“全入口接入”方案，浏览数出现在所有相关入口：

1. 学习样本新建
2. 学习样本生命周期回填
3. PDF 批量导入
4. 平台结果快捷回填
5. 样本池卡片与摘要展示

### 数据结构要求

浏览数字段与现有互动指标统一收纳在同一层级的 `metrics` 中。

也就是所有涉及指标的地方都扩展为：

- `likes`
- `favorites`
- `comments`
- `views`

第一版继续按数值型非负整数处理。

## 设计三：浏览数参与参考样本池判定

### 总体原则

浏览数现在就参与参考样本池判断，但不是主判定指标，而是：

- `辅助放宽指标`

### 判定方式

参考样本池仍然优先看：

- 点赞
- 收藏
- 评论

浏览数只在“接近达标”时补足。

### 为什么不让浏览数单独决定

不采用“浏览数单独直接入池”的原因：

1. 高浏览不等于高质量参考样本
2. 浏览高但互动弱的内容，未必适合作为正向生成与校验参考
3. 直接放开会让参考样本池变得过松

### 推荐口径

第一版规则应满足以下原则：

1. 若 `点赞 / 收藏 / 评论` 已直接达标，则照常进入参考样本池
2. 若三项互动“接近达标”，则允许 `高浏览数` 作为补足信号
3. 若只有浏览高、互动明显偏弱，则先不进入参考样本池

### 前端解释要求

前端必须明确说明样本为什么进入参考样本池，例如：

- `互动达标`
- `互动接近达标，已由高浏览数补足`

这样用户可以分辨：

- 它是靠核心互动入池
- 还是靠浏览数辅助放宽入池

## 设计四：学习样本页的最终信息架构

### 顶层结构

学习样本页最终建议收成三块：

1. 日常处理
2. 扩展维护
3. 样本池弹窗入口

### 日常处理区

保留并组织以下模块：

- 生效流转说明
- 学习样本工具栏
- 学习样本记录列表
- 单条记录详情编辑
- 回流待处理区

其中回流待处理区需要承接原回流中心的高频内容：

- 未处理违规反馈
- 待确认误报
- 待沉淀规则候选

### 扩展维护区

继续保留但收口为低频维护位：

- 规则维护
- 系统校准
- 术语表

### 样本池弹窗

继续保持独立大弹窗，不再往其中塞以下内容：

- 词库
- 回流中心完整列表
- 深度编辑表单

## 测试要求

至少覆盖以下场景：

1. 前端不再保留 `回流中心` 独立 tab
2. 学习样本页能承接回流待处理区
3. 规则维护仍位于学习样本页的扩展维护区
4. 浏览数出现在：
   - 学习样本新建
   - 生命周期回填
   - PDF 批量导入
   - 平台结果回填
5. 浏览数被展示在样本池卡片或摘要中
6. 参考样本池解释文案能区分：
   - 互动达标
   - 高浏览补足
7. 既有参考样本池 / 样本池弹窗 / 样本库去重 / 生成参考筛选链路不被破坏

## 验收标准

完成后应满足：

1. 用户处理样本、回流和平台结果时，主入口统一为 `学习样本`
2. `回流中心` 不再作为独立高频页签存在
3. 词库统一位于 `学习样本 > 扩展维护 > 规则维护`
4. 浏览数作为第四个互动指标全链路可录入、可展示
5. 浏览数会参与参考样本池判断，但只作为辅助放宽，不单独决定入池
6. 前端能清楚解释样本为什么进入参考样本池
